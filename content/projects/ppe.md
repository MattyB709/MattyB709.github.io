# Abstract

This summer I worked at the Eurofins Center of Excellence where I built an end-to-end object detection system for automatic personal protective equipment (PPE) detection. The purpose was to automatically identify whether or not people in laboratories were wearing their lab coat, gloves, and eyewear without the need for manual supervision. I started out by creating my own custom dataset from our laboratory's security camera feed, manually drawing boxes around people and labeling them for their PPE, ending with a dataset of over 1100 images. From here I finetuned [YOLOX](https://github.com/Megvii-BaseDetection/YOLOX) on the new dataset and classes, getting an mAP of 0.56 on a held out test set. I quantized the model down to INT8 precision using post training quantization and optimized with TensorRT for deployment on a Jetson Orin Nano. With INT8 precision, the model achieved a speedup of over 3x when compared to the base PyTorch model (see the image below) with only a slight decrease in mAP.

## Dataset Collection and Model Finetuning

I started out building a pipeline for dataset collection and annotation. The security camera footage was my main source of data, so I built a pipeline to download footage from each of the cameras, extract frames ~15 seconds apart, and import them into [CVAT](https://www.cvat.ai/) for data annotation. In total I had 6 classes: [coat, no-coat, eyewear, no-eyewear, glove, no-glove].  Data annotation was the slowest part of the whole project, in part because there was a lot of ambiguity in how I should label objects. Should a person's whole body be considered "not a coat", or just their torso? If someone was half-covered, by another object, should they still be labeled? These questions had big impacts for model performance. Pretty early on I found that only labeling an object when it was clear what it was reduced false positives, like mistaking a small white box in the corner of an image for a lab coat. 

### YOLOX

YOLO is an object detection model that can both identify and classify objects in an image by drawing bounding boxes. The first YOLO model was released in 2015, and since then there have been many [new generations](https://docs.ultralytics.com/models/yolo11/). The obvious choice here would be to go with the newest generation, YOLOv11, but this was a closed source model released by Ultralytics that I would have needed an enterprise license to use. The newest open source YOLO version that I found was YOLOX, so I ran with this. From their repo I was able to implement their architecture and loss in PyTorch. The YOLOX architecture is comprised of a backbone, neck, and three heads. The backbone's job is feature extraction, where the image is passed through a variety of CNN layers. From here, the outputs go into the neck, which is based on a [Feature Pyramid Network](https://arxiv.org/abs/1612.03144) (FPN). The neck uses CNN layers and residual connections to build 3 different feature maps of the image, each at a different scale. The point of this is to have three different "interpretations" of the image, each with the job of finding objects at different scales, i.e the biggest feature map would have more granularity and thus be better at detecting smaller objects, with the reverse being true for the smallest feature map. This idea is first explored in the FPN paper linked above, but in the final architecture a network called PAFPN is used, which essentially adds more residual connections to better share information between feature maps. These 3 maps are passed into a different head, each with the job of detecting objects at various scales. 

![yolo head](assets/images/yolo_head.png "Image taken from YOLOX paper")

Above, $H\times W$ is the number of "pixels," which should be interpreted as grid cells, in each feature map. These are passed through CNN layers until the desired output shape, $H\times W\times C$, $H\times W \times 4$, and $H\times W \times 1$, where $C$ is the number of classes. Basically, every grid cell of the feature map is responsible for predicting the probability that there is an object inside of it, which gives the 1, the coordinates and dimensions of a bounding box around said object, which gives the 4, and the probability the object belongs to each class, giving $C$. So in total, each grid cell predicts $5+C$ total items, representing all you need to know about an object. Concatenating outputs from all heads gives a total of $8400 \times 5+C$ predictions, or $8400\times 11$ in my case. This is a lot of predictions, and often an object will be in many different grid cells, so you have many cells predicting the same object. The way this is generally handled in object detection is through a post processing technique called Non-Maximum Suppression (NMS), which filters out overlapping boxes based on whichever one is has a higher confidence score. The confidence score is calculated by multiplying the probability of an object by the highest class probability. By doing this, along with filtering out predictions with a low confidence, we get our final boxes.

## Training loop

For training I had to start out by reparameterizing the model to output 6 classes instead of 80, which is what the original YOLOX had. This simply involved going into the model's state dictionary and replacing the output tensors with new ones that outputted the correct number of classes, initializing them like normal. I implemented a custom PyTorch dataset to read in the images and labels from the YOLO format and apply data augmentations. Most of the data augmentations could be done using Torchvision, such as random flip and jitter, but I had to write my own mosaic code. What mosaic does is it stitches together four images randomly, as shown below. 

![Mosaic image](assets/images/mosaic.png "An example of the Mosaic augmentation, done by stitching 4 images together and cropping down to regular size.")

The difficult part about this was correctly shifting all the bounding boxes to their new positions, but once I got it working the results were clear. The two biggest difference makers in model performance (besides pretraining) were dataset size and the data augmentations:

![map vs data](assets/images/mapvsdata.png "Increasing dataset size and applying augmentations led to large increases in mAP. Each point corresponds to the maximum mAP achieved during a 200 epoch training run for a given dataset size. ")

The primary metric for YOLO is mean Average Precision (mAP), which is the area under the Average Precision curve, meant to both measure the model's recall and precision. Interestingly, the data augmentations weren't always helpful, but for the maximum dataset size it made a sizable difference in mAP. 

All training was done on a computer with a 3070 ti. The medium YOLOX model was the biggest model I could train in a reasonable time, even after applying mixed precision training to decrease training time.

## Deployment and Quantization

For initial deployment I was given the Nvidia Jetson Orin Nano, a small, lightweight computer with a GPU on it used primarily for edge AI applications. To optimize the model for performance, I used TensorRT, an Nvidia library that takes a neural network and optimizes it for the current hardware, in my case the nano. There are many optimizations it makes, but two of the biggest ones are layer fusion and reduced-precision/quantization. There are some neural network layers that can be fused into one during inference, such as a convolution layer followed by a batch normalization layer, which can be fused by factoring in the batch norm constants into the convolution layer's weights. For reduced precision, TensorRT changes the model's weights and activations from being represented by a 32 bit floating point number to a 16 bit floating number, speeding up computation with a negligible drop in performance. More interesting is quantization down to 8-bit integers. For INT8 quantization, one popular albeit simple technique is post training quantization (PTQ). This works by running various input images into the model to get a distribution of floating point activations before and after each layer. Using these distributions a scaling factor is computed to scale the floats down into a range representable by INT8, i.e (-128,127). The float is then rounded to an integer value. For a given floating point tensor $x$, the conversion to an integer tensor $q$ can be described as $$q=\text{clip}(\text{round}(\frac x S),-128,127),$$
where $S$ is the scaling factor. The way $S$ is computed depends on implementation and was handled by the [TensorRT model optimizer](https://github.com/NVIDIA/TensorRT-Model-Optimizer) library, which confusingly is not the same as tensorRT. To get back to a float you simply do $x=qS$. By doing this for many layers and inserting linear quantization/dequantization nodes in the model graph to convert activations to INT8 precision and then back up to FP32 precision, much of the computation can be done in INT8 precision. Some of my tested results on the Nano are below:

![map vs quantization](assets/images/mapvquant.png "FPS achieved by different levels of precision set by TensorRT, with the PyTorch model used as a baseline. Lower precision consistently led to higher FPS. All measurements were taken on the Jetson Orin Nano.")
![fps vs quantization](assets/images/fpsvquant.png "The mAP achieved by varying levels of precision set by TensorRT. The only decrease in mAP came from quantizing the model to INT8.")

With lower precision, TensorRT made huge gains in speed with only a negligible drop in performance. This was huge! With this, my part of the project was done, and I was able to hand it off to other engineers to integrate it into the laboratory system.
